# Product

## Register

brand

## Users

Elegant Developer Studio serves people who evaluate Ray as a design engineer, read technical and product writing, inspect selected work, and decide whether there is enough taste, rigor, and reliability to collaborate. They arrive from a public link, GitHub, a shared article, or a project discussion, usually scanning first and drilling into writing, work, knowledge, lab experiments, uses, and contact paths when trust is earned.

## Product Purpose

The site is a public personal studio, not a resume template, GitHub clone, SaaS dashboard, or generic portfolio. It should make aesthetic judgment, engineering credibility, writing depth, project memory, and collaboration readiness visible through real content and working interfaces. Success means a visitor can understand Ray's identity, read durable thinking, inspect evidence, reach every major surface from public navigation, and follow the evolution of the project through GitHub, Vercel, and Feishu records.

## Brand Personality

Warm, technical, and exacting. The site should feel like a quiet design-engineering atelier: editorial enough for long writing, precise enough for source links and command surfaces, personal enough for photos/music/about, and disciplined enough that every page has a durable role.

## Anti-references

- Generic portfolio templates with decorative cards and shallow case studies.
- GitHub profile clones that confuse developer identity with repository chrome.
- High-density SaaS dashboards that turn the homepage into an admin console.
- Fake terminal aesthetics, decorative command UI, and technical surfaces that do not perform real work.
- Mechanical bilingual sites that translate everything without deciding what each language should carry.
- AI-looking brand pages that rely on beige sameness, repeated tiny uppercase section labels, glass decoration, or identical icon-card grids.

## Design Principles

1. Public reachability beats internal completeness: a surface is not real until users can reach it through visible navigation or a public link.
2. Technical style must be earned by function: source reveal, command traces, metadata, evidence links, and reproducible docs matter more than terminal decoration.
3. Writing is the spine: blog, knowledge, projects, lab, and Feishu records should form a readable system rather than isolated pages.
4. Chinese carries judgment and project memory; English remains natural for code, component names, APIs, brands, and open-source context.
5. Restraint must still have a point of view: the site should be calm and elegant, but never anonymous or template-like.

## Accessibility & Inclusion

Target at least WCAG AA for contrast, keyboard access, focus states, semantic navigation, and responsive behavior. Motion must respect `prefers-reduced-motion`. Text and controls must not overflow on mobile. Public paths should be testable by Playwright across desktop and mobile, and every interactive feature that matters should expose visible feedback instead of relying on hidden state.
