# 第四阶段键盘与搜索增强复盘

日期：2026-06-13

## 结论

第四阶段第二个切片已经完成：Command Center 从“可打开、可搜索、可跳转”升级为“可用键盘快速操作、结果可扫读、最近访问可回忆”的全站入口。

这个切片没有把搜索做成完整搜索引擎，也没有引入模糊搜索库。这个选择是正确的。当前站点内容量仍小，真正的问题不是检索召回，而是命令层的手感、信息分组和可恢复路径。

## 已执行内容

- `Cmd K` 打开全站 Command Center。
- `Esc` 关闭。
- `ArrowDown` / `ArrowUp` 移动 active result。
- `Home` / `End` 跳转首尾结果。
- `Enter` 打开 active result。
- 无查询时展示 `Recent`、`Quick actions`、`Writing`、`Projects` 等分组。
- 搜索结果按类型分组。
- 最近访问只记录 command id 和时间。
- 标题和描述出现命中高亮。
- 底部键盘提示提供轻量操作线索。
- `<html>` 增加 `data-scroll-behavior="smooth"`，消除 Next.js smooth-scroll 警告。

## 复盘判断

### 做对的地方

- Command Center 开始具备程序员工具的真实效率，而不是只像一个漂亮弹窗。
- 最近访问让音乐、文章、项目这些高频入口可被“回忆”，符合工作流心智。
- 分组让结果增长后仍能扫描，而不是变成无结构列表。
- 命中高亮只作用在可见文本，不伪造隐藏关键词解释。

### 暴露的问题

- 无查询默认排序仍偏静态，没有理解用户正在什么页面。
- `knowledge`、`lab`、`uses`、`about` 等规划路由没有可解释状态，用户可能误以为搜索失败或页面不存在。
- 空状态还只是“没有结果”，缺少下一步建议。
- 最近访问有可能抢走当前页面上下文分组，导致 `/music` 这类页面反而看不到 `Music context`。

## 追加细节

第三个切片应补齐：

- 上下文感知排序：根据当前路由提升对应内容类型。
- 当前页面上下文分组：例如 `/blog` 默认出现 `Writing context`，`/music` 默认出现 `Music context`。
- 规划中页面提示：例如搜索 `uses` 时说明它属于 Portfolio OS 阶段。
- 空状态恢复建议：提供 `projects`、`writing`、`photos`、`music` 等可执行建议词。
- 不引入完整搜索库，除非内容规模和搜索失败数据证明有必要。

## 对用户建议的执行复核

用户希望“程序员技术风格、能玩、但不要污染”。当前执行路径是：

- 用键盘、命令中心、最近访问和上下文排序表达程序员风格。
- 不在博客正文放宠物或强装饰。
- 不用炫技动画抢阅读注意力。
- 把巧思放在效率、可解释状态和微反馈上。

这个方向应继续保持。

## 后续进入标准

第四阶段下一片可以推进到上下文搜索与空状态恢复；再下一片才考虑：

- `StatusPanel`
- `FilterBar`
- 全站搜索索引
- 命令历史管理

前提是 Command Center 的基础行为稳定。
