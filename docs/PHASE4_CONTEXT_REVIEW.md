# 第四阶段上下文搜索复盘

日期：2026-06-13

## 结论

上下文搜索与空状态恢复切片完成后，Command Center 已经从“全站搜索弹窗”变成“理解当前页面的导航层”。这个方向成立。

但它仍然只解决了“如何找东西”。第四阶段接下来必须补齐“如何看见当前状态”和“如何缩小内容范围”。否则交互层会过度集中在 Command Center，首页、博客列表和项目列表仍缺少足够明确的操作反馈。

## 已执行内容

- `GlobalCommandMenu` 增加当前路径感知。
- `/blog`、`/projects`、`/photos`、`/music` 默认提升对应上下文。
- 搜索评分增加 context boost。
- 搜索 `knowledge` 时显示真实项目结果，并解释公开知识库属于 Phase 5。
- 搜索 `uses`、`about`、`lab` 等规划页面时，给出规划提示。
- 无结果状态显示可点击建议词。
- 无可选结果时结果容器不再声明 `role="listbox"`。
- 修复最近访问吞掉当前页面上下文的问题。
- 追加修复 `CodeBlock` hydration mismatch。

## 做对的地方

- 规划页面没有伪装成已完成页面，路线图状态被明确展示。
- 空状态开始具有恢复能力，而不是把用户留在死路。
- 当前页面上下文分组保住了“我在哪里”的感知。
- 没有引入完整全文搜索库，避免过早复杂化。

## 暴露的问题

- 主页只用 `Shipping` badge 表达状态，缺少更具体的当前工作状态。
- `/blog` 和 `/projects` 的筛选仍是局部临时函数，交互反馈弱。
- 筛选后没有结果数量，不利于用户理解范围变化。
- 筛选清除动作不够显性，后续内容增加后会影响可维护性。

## 追加判断

下一切片不应该继续堆 Command Center 功能。原因：

- Command Center 的核心行为已经稳定。
- 当前内容规模不需要更强搜索算法。
- 用户更需要在页面表面看见“当前状态”和“筛选结果”。
- `StatusPanel` 和 `FilterBar` 都是后续 Phase 5 可复用组件。

## 下一步进入标准

进入 `StatusPanel` 和 `FilterBar` 切片：

- 首页增加小型状态面板：最近在写、最近在做、最近在听。
- `/blog` 和 `/projects` 统一使用可复用 `FilterBar`。
- 筛选条显示结果数量和清除动作。
- 桌面和移动端都不能出现页面级横向溢出。
- 组件必须能继续服务 `/knowledge`、`/photos`、`/lab` 等后续页面。
