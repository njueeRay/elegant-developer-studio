# 信息架构

## 顶层导航

- `/`：工作室首页。
- `/blog`：长文、技术写作和产品思考。
- `/projects`：精选作品、案例研究和已交付工具。
- `/knowledge`：长期知识、引用、代码片段和学习记录。
- `/uses`：工具、技术栈、工作流和发布管线。
- `/lab`：实验、原型、组件、交互证明和质量门禁。
- `/about`：个人介绍、原则、经历、能力边界和协作方式。
- `/photos`：照片档案和视觉现场笔记。
- `/music`：工作室 mix 和 now-playing 记录。
- `/collaboration`：贡献流程、治理规则、创意 backlog 和外部协作入口。
- `/contact`：联系、讨论和项目入口路由。

## 导航原则

- 公开可达优先于内部存在：进入 sitemap 的页面必须在公开 UI 中有入口。
- 主导航承载主信息架构，不再使用只服务单页滚动的 anchor nav。
- 移动端必须提供完整路径，不允许因为桌面导航隐藏而丢失入口。
- 命令面板是加速器，不是唯一导航。
- 任何新增页面都必须同时更新 sitemap、Command Center、公开导航或清晰的二级入口。

## 首页内容模型

首页应保持“精选工作室入口”，不是完整档案馆。

首屏层级：

1. 姓名和身份。
2. 一句话定位。
3. 简短介绍。
4. 当前状态。
5. 命令入口。
6. 工作台或精选亮点。

首屏以下层级：

1. 精选文章。
2. 精选作品。
3. 媒体入口。
4. Knowledge 预览。
5. Uses 预览。
6. Lab 预览。
7. About 预览。
8. 联系入口。

## 页面职责边界

- `Knowledge`：存储长期理解，不做工具清单。
- `Uses`：展示工具和工作流，不承担个人叙事。
- `Lab`：沉淀组件和实验，不承担简历或作品集职责。
- `About`：说明人、原则和合作方式，不替代 Contact 或 Uses。
- `Collaboration`：定义外部协作规则，不混入 About 的个人叙事。

## 命名决策

`Knowledge` 替代 `Notes`。

理由：

- `Notes` 暗示临时、私人、低置信度。
- `Knowledge` 可以容纳笔记、书签、片段、引用和长期学习。
- 对一个设计型工程师来说，它更像公开记忆系统，而不是随手笔记本。

置信度：高。
