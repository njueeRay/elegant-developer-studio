# 第五阶段 Uses 复盘

日期：2026-06-13

## 结论

`/uses` 首版成立。它没有落入工具 Logo 墙，而是把工具、角色、信号、工作流和发布管线放在同一个工作室语境里。

置信度：高。

## 已真正执行的用户建议

- 用户希望“项目地图完善、方便追溯”：`/uses` 已进入 `PROJECT_MAP`、`ROADMAP`、`VERSION_TRACE`、`PROGRESS_LOG`、飞书同步地图和 GitHub issue。
- 用户希望“组件高价值且符合场景”：`UsesExplorer`、工具筛选、`Copy ref`、工作流、发布管线都能继续复用，不是一次性装饰。
- 用户希望“程序员技术风格有巧思”：`Copy all`、`Copy ref`、Command Center 入口、工作流节奏和发布管线比假终端更有用。
- 用户希望“飞书首选中文”：相关本地文档继续用中文作为事实源。

## 成功点

1. 工具不只是名字，而有 `category`、`role`、`description`、`signal` 和稳定 slug。
2. `FilterBar` 再次证明可以跨内容、媒体、知识和工具复用。
3. `Copy ref` 让工具卡能进入 GitHub、飞书、评论和后续文档。
4. Command Center 从规划提示升级为真实 Uses 结果，并支持 `Uses context`。
5. 视觉上使用书架和工作台意象，但没有把页面做成不可维护插画。

## 需要补充但暂不阻塞

- 后续可以给部分工具添加真实链接、配置片段或本地脚本路径。
- 工作流可以继续沉淀为更细的 `WorkflowStep`，支持“输入、动作、输出、追踪位置”。
- 如果 `/lab` 建起来，`/uses` 里的工具应能反向链接到对应实验。
- 如果真实照片质量足够，后续可以替换部分代码原生工作台意象；当前没有必要强行引入低信息密度图片。

## 不建议补的东西

- 不建议做完整设备清单。当前个人站更需要表达“如何工作”，不是炫硬件。
- 不建议加入复杂评分或收藏状态。工具不是商品列表。
- 不建议做假终端或自动打字效果。它们会降低工作室气质。

## 对下一阶段的启发

下一阶段应该做 `/about`。

理由：

- `/knowledge` 说明这个站有公开记忆。
- `/uses` 说明这个站有工作方式。
- `/about` 需要补上“这个人是谁、如何判断、适合怎样合作”。
- `/lab` 更依赖已有组件库存；现在做容易空。
